Phase 1 · Discovery

The documented process is rarely the real process.

Ask, and you'll hear "an email arrives." Watch, and every simple step hides the real work.

01 · The trigger

An email arrives

40+ senders, no two alike. Half are exceptions in disguise. Routing lives in one person's head.

02 · Re-keyed by hand

Copied into a spreadsheet

Which spreadsheet? There are four. A typo doesn't error — it becomes the new source of truth.

03 · Tribal knowledge

Checked in an internal system

No API, no export. "Status" means five things. The one expert retires in eight months.

04 · Exists as a reply

Approved in Slack

The approval of record is a chat message, findable only by the person who asked for it.

05 · 30 required fields

Entered into the ERP

The official system arrives last, fed by everything undocumented that came before it.

If you don't look, you build for a system that doesn't map to reality — which is how most AI pilots fail.

Phase 1 · Discovery

Sit with the work, not the org chart.

Shadow full days

In a one-hour meeting people describe the job they think they have. Beside them for eight, you see the undocumented exception happen.

Get system access

ERP, CRM, the four spreadsheets, the Slack approvals. Read the real artifacts, not the SOP.

Expect multiple owners

Several people run the "same" process differently. One source is a sample size of one.

Chase the exceptions

One way to go right, a thousand ways to go wrong. The unwritten rules are the actual spec.

Discovery is also where trust is built — the relationship is infrastructure.

Phase 2 · Audit

Decide what should be automated — and what should not.

Deterministic software

When rules and inputs are predictable. Most of a good solution is this — don't spend a model on an if-statement.

·
An agent

When the objective is clear but the inputs, path, or actions vary. Often only 3 steps of 10 need judgment.

·
A human in control

When the decision carries material ambiguity, accountability, or irreversible consequences.

Every expected gain lands in one of three buckets: revenue uplift, risk mitigation, cost savings.

Phase 3 · Build

Build for the unhappy paths.

There is one way something goes right and a thousand ways it goes wrong. The agent that handles the exceptions is the one worth deploying.

Build09 / 16
Phase 3 · Build

Add capability in this order.

01

Agent loop

Prompt → model → response → next step, until done or a max-step limit hits.

02

Tool use

Start minimal — one API call, one search. The agent decides when.

03

Guardrails

Input validation · max-step limit · output filtering.

04

Audit trail

Log every prompt, tool call, result, error, timestamp. No visible traces, no trust.

05

Structured outputs

A defined JSON schema, validated on every response.

06

State & recovery

Checkpoint every few steps. Kill it deliberately — it must resume.

07

Failure handling

Enumerate the modes and define behavior for each.

08

Retries & backoff

Exponential on every external call: 1s · 2s · 4s · 8s, capped.

Pick one model and one platform and get genuinely good at it — your value is judgment, not the inference layer.

Phase 5 · Deployment

Make it work inside the business.

01

Integrate with what exists

Never open with "first, migrate off your ERP." Build on top of the current stack — that's the edge over rip-and-replace.

02

Test in a controlled environment

A sandbox inside the company's own infrastructure, where you can run, inspect, and debug safely.

03

Increase autonomy gradually

Shadow mode → approve every action → approve only risky actions → autonomy where earned.

The pattern to push for: agent drafts — a human approves — the system completes the rest.

Phase 5 · Deployment

Deployment is a people problem.

You are a risk to your sponsor

Status quo is safe; bringing you in is not — if it fails, it's their name on it. De-risk them: visible traces, human approval gates, gradual autonomy, measurable checkpoints. Prove value before asking for trust.

Your job is to get them promoted

Drive value cost-effectively in the three buckets and make it attributable, so at review time they can point at it. Face to face makes every hard conversation easier — the relationship is infrastructure.

When production breaks, it's your name on the line — know who to call, what to do, and how to fix it.
