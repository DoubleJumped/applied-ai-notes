Phase 1 · DiscoveryExhibit 4

The documented process is rarely the real process

Ask, and you'll hear "an email arrives." Watch, and every simple step hides the real work.

01 · The trigger

An email arrives

40+ senders, no two alike. Half are exceptions in disguise. Routing lives in one person's head.

02 · Re-keyed by hand

Copied into a spreadsheet

Which spreadsheet? There are four. A typo doesn't error — it becomes the new source of truth.

03 · Tribal knowledge

Checked in an internal system

No API, no export. "Status" means five things. The one expert retires in eight months.

04 · Exists as a reply

Approved in Slack

The approval of record is a chat message, findable only by the person who asked for it.

05 · 30 required fields

Entered into the ERP

The official system arrives last, fed by everything undocumented that came before it.

If you don't look, you build for a system that doesn't map to reality — which is how most AI pilots fail.

Phase 1 · DiscoveryExhibit 5

Discovery happens beside the work, not in a meeting room

Shadow full days

In a one-hour meeting people describe the job they think they have. Beside them for eight, you see the undocumented exception actually happen.

Get system access

ERP, CRM, the four spreadsheets, the Slack approvals. Read the real artifacts, not the SOP.

Expect multiple owners

Several people run the "same" process differently. One source is a sample size of one — sit with more.

Chase the exceptions

One way to go right, a thousand ways to go wrong. The unwritten rules for what counts are the actual spec.

Discovery is also where trust is built — the relationship is infrastructure for everything that follows.

Phase 2 · AuditExhibit 7

Route every step: deterministic software, an agent, or a human

Deterministic software

When the rules and inputs are predictable. Most of a good solution is this — don't spend a model on an if-statement.

·
An agent

When the objective is clear but the inputs, path, or required actions vary. Often only 3 steps of 10 need judgment.

·
A human in control

When the decision carries material ambiguity, accountability, or irreversible consequences.

Some workflows shouldn't be touched at all. Prioritize lengthy, high-volume workflows where the improvement is large enough to matter.

Phase 3 · Build

Build for the unhappy paths

There is one way something goes right and a thousand ways it goes wrong. An agent built for the happy path is worth nothing; the one that handles the exceptions is the one worth deploying.

Source: Varick Agents FDE methodology11 / 18
Phase 3 · BuildExhibit 9

Add capability in a deliberate order

1

Agent loop

Prompt → model → response → next step, until done or a max-step limit hits.

2

Tool use

Start minimal — one API call, one search. The agent decides when to call them.

3

Guardrails

Input validation · max-step limit · output filtering.

4

Audit trail

Log every prompt, tool call, result, error, timestamp. No visible traces, no trust.

5

Structured outputs

A defined JSON schema, validated on every response. Retry or escalate on mismatch.

6

State & recovery

Checkpoint every few steps. Kill it deliberately — it must resume where it stopped.

7

Failure handling

Enumerate the modes — missing data, dead APIs, timeouts, duplicates — and define behavior for each.

8

Retries & backoff

Exponential on every external call: 1s · 2s · 4s · 8s, capped. Escalate when retries run out.

Pick one model and one agent platform and get genuinely good at it — your value is judgment about the workflow, not the inference layer.

Phase 5 · DeploymentExhibit 12

Deployment works with the business, not around it

1

Integrate with what exists

Never open with "first, migrate off your ERP." Build on top of the current stack and connect it — that's the edge over rip-and-replace.

2

Test in a controlled environment

A sandbox inside the company's own infrastructure, where you can run, inspect, and debug safely.

3

Increase autonomy gradually

Shadow mode → approve every action → approve only risky actions → autonomy where earned. Smallest useful action first.

The pattern to push for: agent handles intake, validation, and drafting — a human approves — the system completes the rest.

Source: Varick Agents FDE methodology15 / 18
Phase 5 · DeploymentExhibit 13

Adoption is won by de-risking the people who sponsor you

You are a risk to your sponsor

Status quo is safe; bringing you in is not — if it fails, it's their name on it. De-risk them: visible traces, human approval gates, gradual autonomy, measurable checkpoints. Prove value before asking for trust.

Your job is to get them promoted

Drive value cost-effectively in the three buckets and make it attributable, so at review time they can point at it. Face to face makes every hard conversation easier — the relationship is infrastructure.

When production breaks, it's your name on the line: know who to call, what to do, and how to fix it.
